# Feature: 评论审核与反垃圾

> 状态：Implemented  
> 关联：docx/个人博客-应用场景需求与进阶开发.md §九 `blog-comment-moderation`；constitution.md  
> 依赖：specs/blog-comment-like（一级评论与点赞）；specs/blog-auth-rbac（登录写、ADMIN）；兼容 specs/blog-comment-thread（楼中楼，若已落地）  
> 最后更新：2026-07-14

---

## 1. Problem Statement

开放评论后，刷屏与垃圾内容会污染文章讨论区；现有评论一经发表即对访客可见，缺少敏感词拦截、待审可见性与人工处理手段，也缺少针对发评论的频率约束。

本期交付**评论审核与反垃圾**：评论具备审核状态；命中敏感词进入待审（前台不可见）；ADMIN 可审核通过/拒绝；发评论超频返回明确错误。不做完整风控中台；验证码本期不做。

---

## 2. Success Metrics

| 指标 | 目标值 | 度量方式 |
| --- | --- | --- |
| 敏感词不可见 | 内容命中敏感词的评论对访客前台不可见（待审或等价） | 造含敏感词样本，公开列表断言不出现 |
| 人工可审 | ADMIN 可将待审评论设为通过或拒绝；通过后前台可见，拒绝后仍不可见 | 审核接口 + 公开列表前后对比 |
| 干净评论可用 | 未命中敏感词的评论按默认策略可对前台可见（自动通过） | 正常内容发表后公开列表可命中 |
| 超频可拦 | 同一用户短时内发评论超过阈值时被拒绝且错误明确 | 连续超限请求，断言业务错误（非 5xx） |
| 权限正确 | 非 ADMIN 无法调用审核接口；未登录发评论仍 401 | 角色与无 Token 用例 |

---

## 3. User Stories

- 作为访客，我希望只看到已通过审核的评论，以便不被垃圾内容干扰
- 作为登录用户，我希望正常发表干净评论后仍能较快出现在讨论区，以便交流不被无故拖延
- 作为登录用户，我希望命中敏感词时评论进入待审而非直接刷屏，以便站点保持清洁
- 作为管理员，我希望查看待审队列并通过对/拒绝，以便处理误判与垃圾
- 作为管理员，我希望维护敏感词列表，以便按站点需要调整拦截范围
- 作为系统，我希望限制短时发评频率，以便降低刷屏成本
- 作为登录用户，我不应在未登录或超频时成功写入评论

## 4. Acceptance Criteria

- [x] AC-1: 评论模型具备审核状态，至少支持 **`PENDING`（待审）/ `APPROVED`（通过）/ `REJECTED`（拒绝）**；状态变更仅由系统规则或 ADMIN 审核产生，普通用户不可自改状态
- [x] AC-2: **默认策略**：发表时若内容**未**命中敏感词，状态为 `APPROVED`（自动通过）；若**命中**敏感词，状态为 `PENDING`（写入库但进入待审，不因命中而直接 4xx 拒绝整次请求——除非同时触发其它校验失败）
- [x] AC-3: **公开评论列表**（含楼中楼场景下的根与子回复）仅返回 `APPROVED`；`PENDING` / `REJECTED` 对访客与其它登录用户均不可见；无可见评论时返回空列表，`code=0`
- [x] AC-4: 发表接口成功响应须带上该评论的 **`status`**（或等价字段），以便作者知悉是否待审；待审评论不要求出现在公开列表中的「仅自己可见」扩展（本期不做）
- [x] AC-5: 仅可对 **`APPROVED`** 的评论发表回复（若已启用楼中楼）；父评论为 `PENDING` / `REJECTED` / 不存在时拒绝并返回明确业务错误，不得写入脏数据
- [x] AC-6: **敏感词**：系统维护一份可配置词表；匹配规则至少为对评论正文的**大小写不敏感子串包含**（具体归一化/分词细节由 Plan 固定）；命中任一词即视为命中
- [x] AC-7: **ADMIN 可维护敏感词**：至少支持列表查询、新增、删除（或等价停用）；非 ADMIN 调用返回 403；词条内容非空且有合理长度上限（Plan 固定）
- [x] AC-8: **ADMIN 审核接口**：可按状态筛选待审（至少能列出 `PENDING`）；可将指定评论设为 `APPROVED` 或 `REJECTED`；非 ADMIN 返回 403；目标不存在返回 404；重复审核同一终态须幂等或返回明确业务错误（不得 5xx）
- [x] AC-9: **发评论频率限制**：同一登录用户在滑动或固定时间窗口内创建评论（含回复）超过阈值时拒绝写入，返回统一 JSON 下的明确错误（推荐业务 `code` 与 message 标明超频；**不引入 HTTP 429，除非同期修订 constitution**）；默认阈值由 Plan 给出可配置值（建议起点：每用户每分钟 ≤ 5 次）；正常频率不受影响
- [x] AC-10: 未登录发表评论仍返回 **401**；对草稿/下架文章不可评论的规则与现网一致；权限与状态机校验在 **Service 层**执行
- [x] AC-11: **数据兼容**：既有无状态字段的评论迁移后视为 `APPROVED`，公开列表行为与升级前一致，无需手工洗数
- [x] AC-12: **点赞**：公开可见的仍仅为 `APPROVED` 评论；对不可见评论的点赞/取消若被调用须返回明确业务错误或 404（与「资源对调用方不可见」一致），不得靠隐藏记录继续累计公开计数
- [x] AC-13: **日志与隐私**：应用日志不得打印评论正文全文；允许记录评论 id、文章 id、用户 id、状态变更与是否命中敏感词（布尔或词条 id），避免把用户原话写入日志
- [x] AC-14: 管理端具备待审队列与通过/拒绝操作入口；敏感词管理入口可用；访客端公开评论区不展示待审/拒绝项
- [x] AC-15: 关键路径具备可自动化验收手段（测试或脚本）：至少覆盖自动通过、敏感词待审且公开不可见、ADMIN 通过后可见、拒绝后仍不可见、超频拒绝、非 ADMIN 审核 403、旧数据兼容

## 5. Non-Goals

| 不做 | 原因 |
| --- | --- |
| 完整风控中台（设备指纹、图谱、黑白名单运营台等） | 超出个人博客范围 |
| 验证码（图形 / 短信 / Turnstile 等） | 卡片列为可选；本期不做，可后续增量 |
| 登录失败、公开搜索等全站限流 | 留给 `blog-rate-limit`；本期仅约束**发评论** |
| 分布式限流 / Redis | 须改 constitution；本期进程内或 DB 方案即可（Plan 选型） |
| 发文审核流、操作审计日志中台 | 见 `blog-audit-workflow` |
| 评论编辑、软删占位「该评论已删除」 | 删除策略仍遵循现网 / `blog-comment-thread` |
| 机器学习垃圾分类、第三方内容安全云 API | 超出本期；敏感词表足够 |
| 向作者推送「审核结果」站内信 / 邮件 | 无通知系统；创建响应带 `status` 即可 |
| 修改 JWT / 角色模型 | 复用现有 ADMIN |

---

## 6. Constraints

- 必须遵守 `constitution.md`：业务查询用 JPA / 参数化，禁止 SQL 拼接；不引入 Elasticsearch、Redis、消息队列
- 权限校验在 Service 层：写评论需登录；审核与敏感词管理仅 ADMIN；公开读仅已发布文章下的 `APPROVED` 评论
- **兼容**既有评论与点赞：新增状态字段可空迁移或默认 `APPROVED`；不破坏 `blog-comment-like` / `blog-comment-thread` 的父子与点赞契约（在可见性过滤之上叠加）
- 发评论限流为本期反垃圾的一部分；若后续落地 `blog-rate-limit`，允许将评论限流实现合并进统一限流模块，但**不得**在合并前削弱本 Spec 的 AC-9
- 统一响应格式 `{ code, message, data }`；超频与业务拒绝不得返回 5xx 或堆栈
- 关键路径验收与 PR 说明引用 `blog-comment-moderation` 与 Task 编号

## 7. 附录

### 7.1 术语表

| 术语 | 说明 |
| --- | --- |
| 审核状态 | `PENDING` / `APPROVED` / `REJECTED` |
| 敏感词命中 | 评论正文按约定规则匹配到词表中任一条 |
| 自动通过 | 未命中敏感词时创建即为 `APPROVED` |
| 待审队列 | ADMIN 可筛选的 `PENDING` 评论列表 |
| 发评限流 | 对同一用户创建评论（含回复）的时间窗口频率上限 |
| 公开可见评论 | 状态为 `APPROVED` 且所属文章为已发布的评论 |

### 7.2 与既有 Spec 关系

- 承接 `blog-comment-like` / `blog-comment-thread` Non-Goals 中的「审核流、敏感词、反垃圾」
- 与 `blog-rate-limit`：**本期只做发评论限流**；登录/搜索限流不在范围，避免抢做全站限流 Spec
- 与 `blog-audit-workflow` 隔离：本期不发文审核、不做通用操作日志中台
- 楼中楼若已实现：公开树组装前须过滤非 `APPROVED`；回复目标须为 `APPROVED`

### 7.3 现状基线（开 Spec 时）

| 能力 | 现状 |
| --- | --- |
| 数据模型 | `comments` 含内容与可选 `parent_id`；**无**审核状态字段 |
| 发表 | 登录即可写；写入后立即出现在公开列表 |
| 列表 | 公开返回文章下评论（含楼中楼时父+子）；无状态过滤 |
| 敏感词 | 无 |
| 审核 API | 无 |
| 发评限流 | 无 |
| 角色 | `ADMIN` / 普通用户已存在（`blog-auth-rbac`） |

### 7.4 状态流转（WHAT）

```text
创建
 ├─ 未命中敏感词 → APPROVED
 └─ 命中敏感词   → PENDING
                      ├─ ADMIN 通过 → APPROVED
                      └─ ADMIN 拒绝 → REJECTED
APPROVED / REJECTED ← ADMIN 可再次审核调整（允许从拒绝恢复通过，便于纠错）
```
