# Feature: 登录密码修改

> 状态：Implemented  
> 关联：docx/个人博客-应用场景需求与进阶开发.md §九 `blog-password-change`；constitution.md  
> 依赖：specs/blog-auth-rbac（注册 / 登录 / BCrypt / JWT）；建议在 specs/blog-user-profile 之后（资料页可挂改密入口）  
> 最后更新：2026-07-15

---

## 1. Problem Statement

标准版已支持注册、登录与 BCrypt 存密，`blog-user-profile` 已交付本人公开资料读写与访客端会话，但**明确将「改密独立流程」列为 Non-Goals**。用户一旦泄露或需轮换凭据，只能依赖管理员重置或重新注册，缺少本人主动改密能力，不符合基本账号安全预期。

本期交付**登录用户修改本人密码**：须校验当前密码；新密码满足与注册一致的强度约束并以 BCrypt 落库；错误旧密、未登录、校验失败均有明确业务错误；接口与前端入口不泄露哈希或明文密码。不做邮箱找回、管理员代改密、OAuth；会话吊销留给 `blog-auth-refresh`。

---

## 2. Success Metrics

| 指标 | 目标值 | 度量方式 |
| --- | --- | --- |
| 本人可改密 | 登录用户凭正确旧密可将密码更新为符合规则的新密 | API：改密成功后可用新密登录，旧密登录失败 |
| 旧密校验 | 旧密错误时拒绝改密且原密码不变 | 错误旧密断言业务错误；随后仍可用旧密登录 |
| 强度与一致 | 新密长度/规则与注册一致；确认不一致或非法被拒 | 超短、过长、确认不匹配等用例返回明确错误 |
| 敏感不泄露 | 请求日志与响应不含明文密码或 `passwordHash` | 日志抽查 + 响应断言无 password / hash |
| 未登录拒绝 | 未带有效凭证调用改密返回 401 | API 断言 401 |

---

## 3. User Stories

- 作为登录用户，我希望在验证当前密码后设置新密码，以便定期轮换或怀疑泄露时自行处理
- 作为登录用户，我希望输错当前密码时得到明确提示且原密码不变，以便避免误改
- 作为登录用户，我希望新密码规则与注册时一致，以便行为可预期
- 作为访客，我未登录时不能改任何人的密码，以便账号安全
- 作为系统运营者，我不希望密码明文或哈希出现在日志与 API 响应中，以便满足安全基线

---

## 4. Acceptance Criteria

- [x] AC-1: 提供**需登录**的「修改本人密码」接口；成功时统一 JSON `{ code, message, data }`；`data` 可为 `null` 或仅含非敏感确认信息（如 `userId` / `username`）；**不得**返回 `password` / `passwordHash` / 明文新密
- [x] AC-2: 请求体包含：**当前密码**、**新密码**、**确认新密码**；确认须与新密码一致，否则返回明确业务错误（非 5xx）
- [x] AC-3: **校验当前密码**：与库中 BCrypt 哈希比对；不匹配时返回明确业务错误（文案区分于「未登录」，Plan 锁定为 400「当前密码不正确」），且**不**更新 `password_hash`
- [x] AC-4: **新密码规则**与注册接口一致（长度 6～64，与 `RegisterRequest` 对齐）；非法或超限返回明确业务错误；新密码经 BCrypt 编码后写库
- [x] AC-5: **禁止无意义改密**：新密码与当前密码相同（明文相同）时**必须**拒绝并返回明确业务错误
- [x] AC-6: 改密成功后，用户须能用**新密码**登录成功，用**旧密码**登录失败；`username` / `role` 等其它字段不变
- [x] AC-7: 未登录或 Token 无效调用改密返回 **401**；用户**只能**改自己的密码；本期**不提供**管理员代改任意用户密码的接口；权限在 **Service 层**强制
- [x] AC-8: **前端入口**：已登录用户可从资料页（`/profile`）到达改密表单；提交成功有明确反馈；失败展示后端/校验错误信息；表单字段为 password 类型，不在 UI 明文回显已提交密码
- [x] AC-9: 密码与 Token **全文禁止**写入应用日志；错误响应不回显用户提交的密码值
- [x] AC-10: 关键路径具备可自动化验收手段（测试或脚本）：至少覆盖成功改密后新旧密登录、旧密错误、未登录 401、新密校验失败、响应无敏感字段

---

## 5. Non-Goals

| 不做 | 原因 |
| --- | --- |
| 忘记密码 / 邮箱或短信重置 | 无邮箱体系；留给后续「找回密码」独立 Spec |
| 管理员代设/重置任意用户密码 | 超出本期；仅本人改密 |
| OAuth、第三方登录、修改登录用户名 | 见既有 Non-Goals；用户名仍不可改 |
| Refresh Token、改密后强制全端下线 / 吊销 Access | 见 `blog-auth-refresh`；本期改密后既有 Access JWT 在过期前仍可按现网行为使用；前端不强制重新登录 |
| 登录失败次数锁定、改密接口专用限流中台 | 见 `blog-rate-limit`；可与之后续合并 |
| 密码历史禁止复用 N 次、复杂度评分 UI | 超出个人博客本期范围 |
| 资料字段（展示名/简介/头像）变更 | 已由 `blog-user-profile` 交付 |

---

## 6. Constraints

- 遵守 `constitution.md`：不引入 Elasticsearch、Redis、消息队列、对象存储 SDK、SSR；业务规则在 Service 层强制执行
- 密码仅存 BCrypt 哈希；日志禁止输出密码、Token 全文、连接串密钥
- 统一响应格式 `{ code, message, data }`；兼容既有注册/登录与 JWT claims
- 新密码校验规则须与 `blog-auth-rbac` 注册约束对齐，避免「能注册不能改成同类密码」或反之
- 不破坏 ADMIN 种子账号与其它角色登录；改密不改变角色与文章归属
- 关键路径验收与 PR 说明引用 `blog-password-change` 与 Task 编号

## 7. 附录

### 7.1 术语表

| 术语 | 说明 |
| --- | --- |
| 当前密码 / 旧密码 | 用户改密前用于证明身份的既有登录密码 |
| 新密码 | 通过校验后将写入 BCrypt 哈希的目标密码 |
| 本人改密 | 仅针对当前 JWT 对应用户；无代改他人能力 |

### 7.2 与既有 Spec 关系

- 承接 `blog-user-profile` Non-Goals 中「改密独立流程」
- 复用 `blog-auth-rbac` 的 BCrypt 与登录校验语义；不重做注册/RBAC
- 与 `blog-auth-refresh`：本期不要求改密后服务端吊销 Token；若后续落地 refresh，可再对齐「改密即失效其它会话」
- 与 `blog-rate-limit`：改密/登录防刷可后续叠加，非本期必做

### 7.3 现状基线（开 Spec 时）

| 能力 | 现状 |
| --- | --- |
| 存密 | `User.passwordHash` + BCrypt；注册/登录已有 |
| 改密 API | 无 |
| 资料页 | `/profile` 可编辑展示名/简介/头像；无改密表单 |
| 找回密码 | 无 |

### 7.4 Plan 已锁定（摘要）

| 项 | 锁定值 |
| --- | --- |
| 接口 | `PUT /api/me/password` |
| Body | `currentPassword`、`newPassword`、`confirmPassword`（后端强制校验确认） |
| 新密码长度 | 6～64（与注册一致） |
| 新旧相同 | **强制拒绝** |
| 旧密错误 | `code=400`，`当前密码不正确` |
| 成功 data | `{ userId, username }` |
| 改密后会话 | 不吊销 Access；前端不强制重登 |
| 前端入口 | `/profile` 内「修改密码」区块 |

详见 `plan.md`。
