# Feature: 楼中楼评论

> 状态：Implemented  
> 关联：docx/个人博客-应用场景需求与进阶开发.md §九 `blog-comment-thread`；constitution.md  
> 依赖：specs/blog-comment-like（一级评论与评论点赞）；specs/blog-auth-rbac（登录写）  
> 最后更新：2026-07-14

---

## 1. Problem Statement

标准版 `blog-comment-like` 仅支持文章下的一级评论，读者无法针对某条评论回复，讨论只能平行堆叠，难以还原「谁在回谁」。

本期交付**楼中楼评论**：在兼容既有一级评论数据的前提下，引入 `parentId`（或等价）自关联；登录用户可回复已有评论；公开列表能还原父子关系；回复深度有明确上限且可测；删除策略明确。审核、敏感词与反垃圾留给 `blog-comment-moderation`。

---

## 2. Success Metrics

| 指标 | 目标值 | 度量方式 |
| --- | --- | --- |
| 可回复 | 登录用户可对已有评论发表回复；回复归属正确文章与父评论 | API：带 `parentId` 创建后列表可命中 |
| 父子可还原 | 公开评论列表能还原「父评论 + 子回复」关系，而非仅扁平无关联 | 造 1 父 + ≥2 子样本，断言结构或 `parentId` 归属 |
| 深度上限 | 超过允许层数的回复被拒绝且错误明确 | 对已达深度上限的评论再回复，断言业务错误（非 5xx） |
| 数据兼容 | 既有无 `parentId` 的一级评论仍为根评论，可读可删可赞 | 迁移后旧数据出现在根层；点赞仍可用 |
| 未登录拦截 | 未登录发表回复返回 401 | 无 Token 调用创建回复 |

---

## 3. User Stories

- 作为登录用户，我希望回复某条评论，以便针对观点讨论而非另开平行楼
- 作为访客，我希望在文章下看到根评论及其回复的层级，以便理解对话上下文
- 作为登录用户 / 管理员，我希望删除评论时行为可预期（含是否带走子回复），以便不留下悬空数据
- 作为登录用户，我希望仍能对根评论或回复点赞/取消，以便表达态度
- 作为访客，我不应在未登录时成功发表回复

---

## 4. Acceptance Criteria

- [x] AC-1: 评论模型支持自关联：`parentId`（或等价外键）可空；为空表示一级（根）评论；非空表示回复，且父评论必须属于**同一文章**
- [x] AC-2: 登录用户可对**已发布**文章下的已有评论发表回复：请求携带目标父评论标识；内容非空且长度上限与现有一级评论一致（当前 `content` ≤ 1000）
- [x] AC-3: **深度上限为 2**：仅允许「根评论 → 一层回复」。回复的父评论必须是根评论（`parentId` 为空）；若父评论已是回复，拒绝创建并返回明确业务错误（如 400），不得写入更深层级
- [x] AC-4: 公开获取某文章评论时，返回可还原父子关系的结构：至少为「根评论列表，每项含其子回复列表」；根评论与同层回复均按创建时间**正序**（与现网一级列表一致）；无评论时返回空列表，`code=0`
- [x] AC-5: 不传 `parentId`（或显式空）时行为与现有一级评论创建兼容，新建根评论
- [x] AC-6: **删除策略（级联）**：删除根评论时，级联删除其全部子回复，并清理这些评论上的点赞记录；删除某条回复时仅删除该回复及其点赞，不影响父评论与其它回复。权限不变：评论作者可删自己的，ADMIN 可删任意
- [x] AC-7: 未登录调用发表评论/回复或删除接口返回 **401**；对草稿/下架文章不可评论或回复（404 或与现网一致的业务错误）
- [x] AC-8: 父评论不存在、父评论不属于该文章、或违反深度上限时，不得创建脏数据，返回明确业务错误（非 5xx）
- [x] AC-9: **点赞目标仍含评论**：根评论与回复均可被点赞/取消；幂等与计数规则与 `blog-comment-like` 一致；列表项继续返回 `likeCount` / `likedByMe`（或等价字段）
- [x] AC-10: 既有一级评论数据迁移后 `parentId` 为空，仍作为根评论出现在列表中，无需手工洗数即可正常展示
- [x] AC-11: 访客端详情页评论区支持对根评论发起「回复」、展示子回复列表；未登录点击回复须引导登录或等价拦截（与写接口 401 一致）
- [x] AC-12: 关键路径具备可自动化验收手段（测试或脚本）：至少覆盖创建回复、列表父子结构、深度拒绝、级联删除、未登录 401、旧数据兼容

---

## 5. Non-Goals

| 不做 | 原因 |
| --- | --- |
| 评论审核流、敏感词、验证码、反垃圾 | 见 `blog-comment-moderation` |
| 无限层级或深度 > 2 的嵌套 | 本期锁定深度 2，降低分页与 UI 复杂度 |
| 评论编辑 / @提及通知 / 站内信 | 超出本期 |
| 评论分页、游标、按热度排序 | 保持现网「单文全量列表」量级；若数据量被迫分页须另开 Spec |
| 删除后「该评论已删除」占位文案 | 本期采用级联硬删，不做软删占位 |
| 修改点赞目标类型枚举或文章点赞行为 | 仅确保评论（含回复）仍可赞 |
| 引入 Redis / MQ 做评论推送 | 须改 constitution；非本期 |

---

## 6. Constraints

- 必须遵守 `constitution.md`：业务查询用 JPA / 参数化，禁止 SQL 拼接；不引入 Elasticsearch、Redis、消息队列
- **兼容** `blog-comment-like` 已有一级评论与点赞：迁移可空 `parent_id`（或等价），禁止破坏既有根评论读写与 `COMMENT` 点赞唯一约束
- 权限校验在 Service 层：写操作需登录；删除作者本人或 ADMIN；公开读仍仅针对已发布文章
- 列表组装父子关系须在服务端完成（邻接表存、树或「父 + 子」出），避免把无序扁平列表甩给前端自行猜层级
- 关键路径验收与 PR 说明引用 `blog-comment-thread` 与 Task 编号

## 7. 附录

### 7.1 术语表

| 术语 | 说明 |
| --- | --- |
| 根评论 / 一级评论 | `parentId` 为空的评论，直接挂在文章下 |
| 回复 | `parentId` 指向某条根评论的评论 |
| 深度 2 | 仅两层：根 → 回复；不允许回复再挂回复 |
| 楼中楼 | 以父子关系展示的评论回复结构（本期深度 2） |
| 级联删除 | 删根评论时同步删除其全部子回复及相关点赞 |

### 7.2 与既有 Spec 关系

- 承接并**撤销** `blog-comment-like` 中「无楼中楼 / AC-6」的约束；一级评论能力保留并扩展为可回复
- 与 `blog-comment-moderation` 隔离：本期不引入审核状态或敏感词
- 点赞契约延续 `blog-comment-like`：`targetType=COMMENT` 对根与回复均有效

### 7.3 现状基线（开 Spec 时）

| 能力 | 现状 |
| --- | --- |
| 数据模型 | `comments`：`id, article_id, user_id, content, created_at`；无 `parent_id` |
| 列表 | `GET /api/articles/{id}/comments` 扁平正序 |
| 发表 | `POST /api/articles/{id}/comments` 仅一级；无 `parentId` |
| 删除 | `DELETE /api/comments/{id}` 硬删单条 + 清该条点赞 |
| 点赞 | `COMMENT` 目标类型已支持 |
