# Feature: 搜索增强（标准版后增量）

> 状态：Implemented  
> 关联：docx/个人博客-应用场景需求与进阶开发.md §九 `blog-search-enhance`；constitution.md  
> 依赖：specs/personal-blog（MVP）；specs/blog-content-enhance（摘要字段 `summary`、正文 Markdown、公开仅已发布）  
> 最后更新：2026-07-14

---

## 1. Problem Statement

MVP 公开列表仅支持**标题**关键词 `LIKE` 匹配。文章增多后，读者常按正文或摘要中的术语检索，当前无法命中，只能靠分类/标签间接筛选。

本期交付**搜索增强**：公开文章检索在保持现有查询参数与分页行为的前提下，将关键词匹配范围扩展为**标题 + 摘要 + 正文**；空关键词与无结果行为明确；结果仍仅包含公开可见的已发布文章。不引入 Elasticsearch。

---

## 2. Success Metrics

| 指标 | 目标值 | 度量方式 |
| --- | --- | --- |
| 字段覆盖 | 仅出现在摘要或正文（标题不含）的关键词可被检索到 | 准备 ≥3 篇样本：词只在标题 / 只在摘要 / 只在正文，分别断言命中 |
| 可见性 | 检索结果中无草稿、已下架文章 | 同词写入草稿与已发布各 1 篇，公开搜索仅命中已发布 |
| 空结果 | 无匹配时返回空分页列表，不报错 | `keyword` 使用不可能命中的字符串，断言 `code=0` 且列表为空 |
| 兼容性 | 原有 `categoryId` / `tagId` / 分页 / 排序与无 keyword 列表行为不变 | 回归：无 keyword 列表与带分类筛选用例 |
| 响应时间 | 本地约定数据量（如 ≤200 篇已发布）下单次公开搜索可接受 | 手工或脚本连续请求，主观/计时参考 300ms 量级（非生产 SLA） |

---

## 3. User Stories

- 作为访客，我希望用关键词搜索标题、摘要或正文，以便快速定位文章
- 作为访客，我希望无匹配时看到空列表而不是错误，以便继续浏览
- 作为访客，我不应通过搜索看到草稿或已下架文章
- 作为访客，我希望搜索可与分类/标签筛选组合使用，以便缩小范围

---

## 4. Acceptance Criteria

- [x] AC-1: 公开文章列表接口在提供非空 `keyword` 时，匹配范围覆盖**标题、摘要、正文**三者：任一字段包含关键词（部分匹配、大小写不敏感）即命中
- [x] AC-2: 同一 `keyword` 下，仅返回公开可见的已发布文章（与现有公开列表可见性规则一致，含状态与发布时间约束）；草稿、已下架不出现
- [x] AC-3: `keyword` 为空或未传时，行为与当前「不按关键词过滤」一致，不因本期变更而改变默认列表
- [x] AC-4: 有 `keyword` 但无任何命中时，返回空列表（分页结构完整），`code=0`，不抛 5xx
- [x] AC-5: `keyword` 可与现有 `categoryId`、`tagId`（及已有公开列表其它筛选，如归档年月若存在）**同时生效**（AND）；排序规则与无 keyword 时的公开列表一致（如置顶优先再按发布时间）
- [x] AC-6: 列表项仍返回既有列表字段（含摘要展示所需信息）；本期**不要求**搜索高亮片段字段
- [x] AC-7: 访客端搜索入口文案/占位提示反映「可搜标题与正文」（或等价说明），避免仍写「仅搜索标题」
- [x] AC-8: 管理端文章列表的关键词行为若已存在，本期**不强制**与公开端对齐；若实现时一并扩展，须在 Plan 中说明且不得削弱公开端 AC

---

## 5. Non-Goals

| 不做 | 原因 |
| --- | --- |
| Elasticsearch / 外部搜索引擎 | 留给 `blog-fulltext-es`；须先改 constitution |
| 搜索建议、热搜、搜索历史 | 超出本期 |
| 相关推荐、同义词、分词纠错 | 超出本期 |
| 结果高亮 HTML 片段字段 | 可选体验，可作后续微小或下期 Spec |
| 独立 `/api/search` 新资源（除非 Plan 证明必要） | 优先扩展现有公开文章列表的 `keyword` |
| Redis 搜索缓存 | 须改 constitution；非本期 |

---

## 6. Constraints

- 必须遵守 `constitution.md`：禁止引入 Elasticsearch、Redis；禁止 SQL 字符串拼接，使用 JPA Specification / 参数化查询
- 必须兼容现有公开列表 API 的路径、查询参数名（至少保留 `keyword`）与统一响应 `{ code, message, data }`
- 公开可见性规则不得弱于 `blog-content-enhance`（已发布且按现有发布时间可见性）
- 关键词匹配为子串包含即可，不要求中文分词器
- 关键路径须有可自动化验收手段（脚本或测试），PR 说明引用 `blog-search-enhance` 与 Task 编号
