# 总纲 · 项目技术规范文档

> 基于 **Spec-Driven Development（SDD）** 方法论，将规格说明作为唯一真实来源  
> （Single Source of Truth），代码作为其派生产物。
> **核心原则：人定义 WHAT，AI 实现 HOW。**

---

## 一、方法论概述

### 1.1 定义

| 项 | 说明 |
| --- | --- |
| **方法名称** | Spec-Driven Development（SDD，规格驱动开发） |
| **核心思想** | Spec 是唯一真实来源；代码、测试、PR 均为 Spec 的派生产物 |
| **人机分工** | 人负责定义问题、边界与成功标准；AI 负责架构方案起草与代码实现 |
| **适用场景** | 需长期维护、多人协作、AI 辅助开发的中大型项目 |

### 1.2 与相关概念的关系

- **不是瀑布模型**：Spec 是活文档，支持增量更新；迭代粒度为**单个功能模块**，而非整个项目
- **与 Vibe Coding 的关系**：探索阶段可快速试错；一旦决定正式开发，立即切换 SDD；所有变更先改 Spec 再改代码
- **成熟度光谱**：
  - **L1 Spec-First**：编码前写 Spec（当前主流）
  - **L2 Spec-Anchored**：Spec 与代码持续同步，测试强制一致性
  - **L3 Spec-as-Source**：人只编辑 Spec，代码完全由 AI 生成维护（目标愿景）

## 二、开发阶段与产出

### 2.1 四阶段流程

```text
Specify（规格定义） → Plan（方案规划） → Implement（代码实现） → Validate（验证确认）
```

| 阶段 | 主导者 | 核心产出 | 关键动作 |
| --- | --- | --- | --- |
| **Specify** | 人 | `spec.md` | 定义问题、边界、成功标准、验收条件 |
| **Plan** | 人 + AI | `plan.md` | 架构选型、模块划分、接口定义；AI 起草，人审核 |
| **Implement** | AI | 代码 + 测试 | 按 `plan.md` / `tasks.md` 逐任务实现 |
| **Validate** | 人 + AI | 测试报告 | 自动化测试 + 人工 Code Review |

**流程说明：**

- Specify 与 Plan 之间可多轮迭代（建议 **3～5 轮** 直至 Spec 合格）
- Implement 与 Validate 持续循环，直至验收通过
- 发现新需求或 Bug：**立即写增量 Spec**，再委派实现，禁止绕过 Spec 直接改代码（重大变更）

### 2.2 项目级时间线参考（7 日 MVP 模型）

| 阶段 | 时间 | 动作 | 产出 |
| --- | --- | --- | --- |
| **DAY 0** | 第 1 天 | 定义 MVP 边界、拆解模块、撰写 Spec、汇入 Wiki | 零行代码；完整 Spec 体系 |
| **DAY 1-2** | 第 2-3 天 | 架构与框架并行开发 | 系统骨架、`plan.md`、`tasks.md` |
| **DAY 3-4** | 第 4-5 天 | Spec 迭代、增量需求 | 增量 `spec.md`、PR |
| **DAY 5-6** | 第 6-7 天 | 打磨、Dogfooding、修复 | 增量 Spec、修复 PR |
| **DAY 7** | 第 8 天 | 发布上线 | 可交付产品 |

## 三、文档体系

### 3.1 文件清单

| 文件 | 层级 | 作用 | 变更频率 |
| --- | --- | --- | --- |
| `constitution.md` | 项目级 | 不可违背的项目原则与全局约束 | 极少变更 |
| `spec.md` | 功能/模块级 | 需求规格，唯一真实来源 | 随需求迭代 |
| `plan.md` | 功能/模块级 | 基于 Spec 的技术方案 | Spec 变更后同步 |
| `tasks.md` | 功能/模块级 | 可执行的原子任务清单 | Plan 变更后同步 |

**目录建议：**

```text
project-root/
├── constitution.md          # 项目宪法（根目录，全局唯一）
├── specs/
│   └── {module-name}/
│       ├── spec.md
│       ├── plan.md
│       └── tasks.md
└── ...
```

### 3.2 constitution.md — 项目宪法

**定位：** 所有 Spec 必须遵守的不可变约束；避免在每个 Spec 中重复声明通用规则。

**格式要求：**

```markdown
# Project Constitution

## 1. 技术栈约束
- 语言 / 框架 / 运行时（固定选型及版本）
- 禁止引入的依赖或技术

## 2. 架构原则
- 分层规范、模块边界、依赖方向
- 接口风格（如 REST、统一响应格式）

## 3. 代码规范
- 命名约定、目录结构
- 必须编写单元测试的模块类型
- 代码重复、抽象层级要求

## 4. 安全约束
- 必须使用参数化查询，禁止 SQL 拼接
- 敏感数据禁止出现在日志中
- 认证 / 授权基本要求

## 5. 质量门禁
- 测试覆盖率下限
- PR 合并前必须通过的检查项
- Code Review 要求

## 6. 文档与 Spec 规范
- Spec 文件命名与存放位置
- 重大变更 vs 微小变更的判定标准
```

## 四、spec.md — 需求规格

### 4.1 写作原则

| 原则 | 说明 |
| --- | --- |
| **只写 WHAT，不写 HOW** | 描述做什么、为什么做；实现方案留给 `plan.md` |
| **可测试** | 成功标准必须可量化、可验证，禁止「很快」「美观」等模糊表述 |
| **技术栈无关** | 换一个技术栈实现，Spec 仍然成立（Constraints 除外） |
| **边界清晰** | Non-Goals 明确列出本期不做的事项 |

**粒度检验标准：**

> 用不同技术栈实现这个 Spec，Spec 是否仍然有效？  
> 若否，说明混入了 HOW，需回退到 WHAT 层。

### 4.2 六要素（必填）

| 要素 | 作用 | 示例 |
| --- | --- | --- |
| **Problem Statement** | 为什么做 | 「当前系统不支持细粒度权限控制」 |
| **Success Metrics** | 做到什么程度算完成 | 「P95 < 50ms，覆盖 20+ 权限类型」 |
| **User Stories** | 谁在什么场景下用 | 「作为管理员，我可以创建自定义角色」 |
| **Acceptance Criteria** | 怎么验证 | 「单用户多角色，权限取并集」 |
| **Non-Goals** | 什么不做 | 「本期不做跨组织权限委托」 |
| **Constraints** | 技术约束（外部限制） | 「必须兼容现有 OAuth2.0 流程」 |

### 4.3 格式模板

```markdown
# Feature: {功能/模块名称}

> 状态：Draft | Review | Approved | Implemented  
> 关联：{issue / PR / 模块路径}  
> 最后更新：YYYY-MM-DD

## 1. Problem Statement
{描述当前问题与业务背景，1～3 段}

## 2. Success Metrics
| 指标 | 目标值 | 度量方式 |
| --- | --- | --- |
| {指标名} | {可量化目标} | {如何测量} |

## 3. User Stories
- 作为 {角色}，我希望 {行为}，以便 {价值}
- ...

## 4. Acceptance Criteria
- [ ] AC-1: {可验证的验收条件}
- [ ] AC-2: {可验证的验收条件}
- ...

## 5. Non-Goals
- 本期不做 {事项}，原因：{简要说明}
- ...

## 6. Constraints
- 必须 {约束条件}
- 禁止 {约束条件}
- 依赖 {已有系统/接口}

## 7. 附录（可选）
### 7.1 术语表
### 7.2 参考链接
### 7.3 变更记录
| 版本 | 日期 | 变更说明 | 作者 |
| --- | --- | --- | --- |
| v1.0 | YYYY-MM-DD | 初稿 | |
```

### 4.4 好 Spec vs 坏 Spec

**坏 Spec 示例（禁止）：**

```markdown
系统需要一个快速的搜索功能。搜索结果应该相关且准确。
```

问题：模糊（「快速」无量化）、遗漏边界、缺乏理由、混入 HOW。

**好 Spec 示例（推荐）：**

```markdown
## Problem Statement
用户在文章超过 500 篇后无法通过分类快速定位内容，需要全文检索能力。

## Success Metrics
| 指标 | 目标值 | 度量方式 |
| --- | --- | --- |
| 搜索响应时间 | P95 < 200ms | 压测 1000 次取分位 |
| 召回率 | Top-10 结果含目标文章 ≥ 90% | 人工标注 100 条查询 |

## Acceptance Criteria
- [ ] AC-1: 支持标题与正文全文检索
- [ ] AC-2: 支持模糊匹配（编辑距离 ≤ 2）
- [ ] AC-3: 无结果时返回空列表，不抛异常

## Non-Goals
- 本期不做搜索结果高亮
- 本期不做搜索历史与推荐

## Constraints
- 必须使用现有 PostgreSQL 实例，不引入 Elasticsearch
```

## 五、plan.md — 架构方案

### 5.1 定位

基于已 Approved 的 `spec.md` 生成的技术实现方案。通常由 **AI 起草、人审核修改**。

### 5.2 格式模板

```markdown
# Plan: {功能/模块名称}

> 基于：specs/{module-name}/spec.md v{X.Y}  
> 状态：Draft | Review | Approved  
> 最后更新：YYYY-MM-DD

## 1. 方案概述
{一段话说明整体思路}

## 2. 架构设计
### 2.1 模块划分
{模块名} — {职责}

### 2.2 数据模型
{表结构 / 实体 / 字段说明}

### 2.3 接口定义
| 方法 | 路径 | 说明 | 请求/响应要点 |
| --- | --- | --- | --- |
| GET | /api/... | | |

### 2.4 关键流程
{时序或流程说明，可用 Mermaid}

## 3. 技术选型
| 决策点 | 选型 | 理由 |
| --- | --- | --- |
| {问题} | {方案} | {原因} |

## 4. 风险与备选方案
| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |

## 5. 与 Constitution 的对齐检查
- [ ] 符合技术栈约束
- [ ] 符合安全约束
- [ ] 符合架构原则

## 6. 变更记录
| 版本 | 日期 | 变更说明 |
| --- | --- | --- |
```

## 六、tasks.md — 任务清单

### 6.1 定位

将 `plan.md` 拆解为**可独立验证的原子任务**，每个任务对应一个可交付、可验收的增量。

### 6.2 格式模板

```markdown
# Tasks: {功能/模块名称}

> 基于：specs/{module-name}/plan.md v{X.Y}  
> 状态：Todo | In Progress | Done

## 任务列表

### Task-1: {任务标题}
- **描述**：{具体做什么}
- **交付物**：{文件/接口/组件}
- **验收标准**：{如何确认完成，对应 spec AC 编号}
- **依赖**：无 | Task-N
- **状态**：[ ] Todo  [ ] Done

### Task-2: {任务标题}
...

## 执行顺序
Task-1 → Task-2 → Task-3（可并行的任务标注 ⫽）
```

**任务拆分原则：**

- 每个任务可在 **1 次 AI 会话 / 1 个 PR** 内完成
- 每个任务必须有明确的验收标准，且可映射到 `spec.md` 的 Acceptance Criteria
- 任务间依赖关系必须显式声明

## 七、Specify ↔ Plan 迭代规范

### 7.1 推荐迭代循环

```text
编写 spec.md（v1）
    → AI 生成 plan.md（v1）
    → 人审视 plan，反向检查 spec 遗漏/模糊点
    → 修订 spec.md（v2）
    → 重新生成 plan.md（v2）
    → … 重复 3～5 轮 …
    → Spec Approved → Plan Approved → 生成 tasks.md → 进入 Implement
```

### 7.2 Spec 合格判定

- [ ] 六要素完整且无空项
- [ ] Success Metrics 均可量化
- [ ] Acceptance Criteria 均可转化为测试用例
- [ ] Non-Goals 已明确
- [ ] 通过「技术栈无关」粒度检验
- [ ] 与 `constitution.md` 无冲突

---

## 八、Implement 与 Validate 规范

### 8.1 实现要求

- AI 按 `tasks.md` 顺序逐任务实现，**禁止跳过 Spec/Plan 擅自扩展**
- 每个任务完成后提交 PR，PR 描述须引用对应 Spec 与 Task 编号
- 代码风格、安全约束以 `constitution.md` 为准

### 8.2 验证要求

| 验证层 | 执行者 | 内容 |
| --- | --- | --- |
| **自动化测试** | AI + CI | 单元测试、接口测试；AC 映射为测试用例 |
| **人工 Review** | 人 | 逻辑正确性、安全性、性能、Spec 符合度 |
| **Spec 符合度** | 人 | 实现是否满足全部 AC，是否触犯 Non-Goals |

**重要：Spec 替代的是需求文档，不是 Code Review。Validate 不可省略。**

### 8.3 增量变更流程

| 变更类型 | 判定标准 | 流程 |
| --- | --- | --- |
| **重大变更** | 可能影响其他模块行为、接口、数据结构 | 写增量 Spec → 更新 Plan → 更新 Tasks → 实现 → 验证 |
| **微小变更** | 纯局部修改（文案、样式、不影响行为） | 可直接修改代码，无需完整 Spec 流程 |

## 九、禁止事项与常见陷阱

| 陷阱 | 症状 | 规范要求 |
| --- | --- | --- |
| **过度规格化** | Spec 比代码还长，规定每个实现细节 | 遵守 WHAT/HOW 分离；使用粒度检验标准 |
| **规格腐烂** | 代码已迭代，Spec 仍停留旧版 | 代码变更与 Spec 变更同步；重大 PR 必须更新 Spec |
| **规格官僚化** | 改按钮颜色也走全流程 | 区分重大/微小变更，微小变更直接改 |
| **虚假信心** | 有 Spec 就放松 Review | Validate 阶段不可跳过；安全约束写入 Constitution |
| **工具过度** | 工具链配置时间超过写 Spec | 最低配置：`constitution.md` + `spec.md` + AI Agent |

## 十、快速启动 Checklist

### 新项目

- [ ] 创建 `constitution.md`，定义技术栈、安全、质量门禁
- [ ] 定义 MVP 边界，列出模块清单
- [ ] 为每个模块创建 `specs/{module}/spec.md`
- [ ] 完成 3～5 轮 Spec ↔ Plan 迭代
- [ ] 生成 `plan.md` 与 `tasks.md`
- [ ] 按 Task 逐条委派 AI 实现
- [ ] 每条 Task 完成后执行 Validate

### 单个功能（增量）

- [ ] 判断重大变更 or 微小变更
- [ ] 重大变更：创建/更新 `spec.md` → `plan.md` → `tasks.md`
- [ ] AI 实现 → 测试 → 人工 Review
- [ ] 合并后更新 Spec 状态为 Implemented
